# Design — Categorisation

*Status: conceptual design for review*

Replaces the placeholder "10 broad categories + Ollama classifier" plan with a
**deep, continually-evolving, low-bias** categorisation system that fits the 8 GB
VPS (no big on-box LLM, no VPS upgrade). Built **on top of clusters**, so it depends
on the clustering fix landing first (`docs/clustering-fix-spec.md`).

## Guiding principles (from CONCEPT.md)

- **Agnosticism / anti-bias** → the evolving taxonomy is **fully automatic**, gated
  only by *algorithmic* quality thresholds and corrected only by *factual* flags —
  never by human taste.
- **Transparency** → every label carries its evidence; methodology and thresholds
  are public; taxonomy changes produce a changelog.
- **Progressive depth** → one small *fixed* anchor + one *emergent* hierarchy.
- **Self-organising, not self-training** → the intelligence lives in the embedding
  space, not in a model's frozen weights.

## The core idea: fixed anchor + emergent hierarchy

Two axes people usually conflate:

1. **Broad categories — curated, stable, ~12.** The onboarding/navigation anchor.
   Must not shift under users. The *only* hand-maintained layer.
2. **Everything below — emergent, automatic, evolving.** Discovered from the
   embedding space; grows and reorganises as the corpus grows.

### Broad categories (the anchor)

- Defined by a **prototype**: a natural-language description ("government, elections,
  legislation, parties, public policy…") plus a few exemplar articles → a **centroid
  embedding**.
- A cluster is assigned by **top-k cosine similarity to the category centroids**
  (not argmax — categories are non-exclusive per CONCEPT). Cheap, multilingual (via
  bge-m3), and **auditable** ("Politics 0.71, World 0.63").
- A small LLM is a **tie-breaker only** on low-margin cases, batched — never the
  classifier.
- **Gentle adaptation, zero training:** each centroid **drifts toward its
  high-confidence members** over time (prototype = seed exemplars + accumulated
  confident members). Recognisable but tracks real usage.
- **Bias posture:** this small set is the one residual editorial choice. Keep it
  conventional, publicly documented, and *soft* (top-k, not a hard gate). Option on
  the table: derive it from the top-level cut of the topic tree to remove even that
  choice — deferred, because onboarding wants a stable, intuitive anchor.

### Emergent hierarchy (the evolving depth)

- **Topic discovery:** a BERTopic-style pass over **cluster centroids** (UMAP →
  HDBSCAN → c-TF-IDF for terms). Finds coherent niches automatically and produces
  new ones as coverage shifts — genuinely continually evolving. HDBSCAN's noise
  handling + a min-size gate keep junk out *algorithmically*.
- **Labels:** generated by the small/occasional LLM from top terms + exemplar
  clusters, stored **with their evidence** for auditability.
- **Hierarchy as cuts, not hardcoded layers:** the model yields a dendrogram;
  "Theme" (mid) and "Niche" (fine) are **cuts at different depths of one tree**, not
  two separately-engineered taxonomies. Presentation:
  **Broad (fixed) → Theme (emergent) → Niche (emergent) → free-text queries.**
  Depth can increase as the corpus grows **without a schema change**.
- **Lifecycle:** topics grow, split, merge, and decay — reusing the cluster
  lifecycle pattern one level up.
- **Cross-category (DAG):** a niche links to *whatever* broad categories its members
  map to ("AI regulation" under Technology *and* Politics), not a strict tree.

### User free-text niches (the long tail)

- A user types "monetary policy" → embed → surface clusters/topics by similarity.
  These are *queries*; frequently-used ones are **promoted** to persistent niches.
  Infinite granularity without predefining everything.

## Categorise at the cluster level

Confirmed decision. A **cluster** is a story (the value unit users browse); a
**topic/category** is a subject. Compute a cluster **centroid** (mean of member
embeddings), assign broad categories + topics to the *cluster*, and let member
articles inherit. Cheaper and cleaner than per-article, and it's the level users
navigate. (This is why the clustering fix is a prerequisite — categorising 21k
mostly-singleton clusters would be meaningless.)

## Topic stability & lineage (the sleeper requirement)

If topics merge/split freely, a user "following AI regulation" must not silently lose
their follow. Requires **stable topic IDs with a lineage/changelog** across
merges/splits — the same problem already solved for entity resolution, and it feeds
the public methodology changelog. Design it in from the start.

## Data model (sketch)

- `settings` — thresholds (shared with clustering): coherence, min topic size,
  merge/split distances, top-k, drift rate. Typed key/value.
- `categories` — curated broad set: name, slug, description, centroid vector,
  exemplar ids.
- `topics` — self-referential `parent_id` (hierarchy), `depth`, centroid vector,
  label, **label evidence** (terms + exemplars), lifecycle status, broad-category
  link(s).
- `topic_membership` — `cluster_id ↔ topic_id`, many-to-many, with a
  similarity/confidence score.
- `topic_lineage` — merge/split history for stable follows + changelog.

## Compute cadence

- **Incremental (cheap, per cluster pass):** assign new/updated clusters to existing
  broad categories + topics by nearest centroid.
- **Batch (periodic, off-peak):** topic **re-fit** — discover new topics, reorganise
  the tree, recompute centroids, emit lineage. Celery Beat.
- **RAM discipline:** run the batch re-fit as an **ephemeral** job (load → compute →
  exit) so UMAP/HDBSCAN memory isn't held resident alongside bge-m3 and Postgres.

## Embedding model

Uses **bge-m3** from the clustering fix (multilingual, 1024-dim, quantized). One
substrate for dedup, clustering, categorisation, and topic discovery. Multilingual
means categorisation works in *any* language against the *same* topic space.

## What "self-learning" honestly means here

No weight training on-box. The system evolves via: (a) broad-centroid drift,
(b) emergent topics from a growing corpus, (c) user-niche promotion, (d) periodic
re-fit reorganising the tree. The one place weight-level learning helps later is
**fine-tuning the embedding model** on-domain (rented GPU, a few hours) — which
improves dedup, clustering, *and* categorisation at once. That, not a bigger on-box
LLM, is the real "learning" lever.

## Relationship to the (frozen) digest

Once **broad categorisation on clusters** works, the digest's category groups
populate and we **unfreeze the digest** — with "Top stories" retained as a
category-independent section. This is the last step, deliberately built on a system
that actually has categories rather than papering over the gap.

## Open decisions (need your steer)

1. The **broad category list** (~12) — define and publicly document (this is a
   product/editorial call, kept deliberately conventional).
2. **Theme / Niche cut depths** — where to slice the dendrogram for presentation.
3. **Label model** — small local LLM (fits 5 GB) vs occasional cloud call for
   higher-quality labels (infrequent → cheap).
4. Initial **coherence / min-size / merge thresholds** — seed values, then calibrate.

## Overall sequencing

1. **Clustering fix** (`docs/clustering-fix-spec.md`), incl. the bge-m3 upgrade.
2. **Broad categorisation** at cluster level (centroids + top-k + drift).
3. **Emergent topic layer** + lineage + labels.
4. **Unfreeze the digest** on real categories (+ Top stories).
